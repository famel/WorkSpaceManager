**Πρωτεύων Actor**: Εξωτερικό Σύστημα (π.χ. Σύστημα HR της Τράπεζας)

**Δευτερεύοντες Actors**: 
- Σύστημα Κρατήσεων (ως ο πάροχος του API)
- Διαχειριστής Συστήματος (για τη διαχείριση της πρόσβασης στο API)

**Προϋποθέσεις**: 
- Το εξωτερικό σύστημα έχει λάβει τα απαραίτητα credentials (π.χ. API Key, OAuth2 Client ID/Secret) από τον διαχειριστή για να έχει πρόσβαση στο API.
- Η IP διεύθυνση του εξωτερικού συστήματος είναι στη λίστα επιτρεπόμενων (whitelist), αν απαιτείται.

**Μετα-συνθήκες**:
- **Επιτυχής**: 
  - Το εξωτερικό σύστημα ανακτά ή αποστέλλει δεδομένα με επιτυχία (π.χ. λαμβάνει μια λίστα με τις σημερινές κρατήσεις).
  - Μια ενέργεια εκτελείται με επιτυχία (π.χ. δημιουργείται μια κράτηση εκ μέρους ενός χρήστη).
- **Αποτυχής**: 
  - Η κλήση στο API αποτυγχάνει λόγω σφάλματος αυθεντικοποίησης, δικαιωμάτων ή λανθασμένων παραμέτρων.
  - Το εξωτερικό σύστημα λαμβάνει ένα μήνυμα σφάλματος με τον κατάλληλο κωδικό κατάστασης HTTP (π.χ. 401, 403, 400).

**Βασική Ροή (Ανάκτηση Δεδομένων)**:
1. Το εξωτερικό σύστημα (client) χρειάζεται να ανακτήσει δεδομένα, π.χ., όλα τα check-ins που έγιναν σήμερα.
2. Το εξωτερικό σύστημα δημιουργεί ένα HTTP αίτημα (request) προς το κατάλληλο endpoint του API, π.χ., `GET /api/v1/bookings?date=today&status=checked-in`.
3. Στο header του αιτήματος, το εξωτερικό σύστημα περιλαμβάνει το token αυθεντικοποίησης (π.χ. `Authorization: Bearer <API_KEY>`).
4. Το Σύστημα Κρατήσεων λαμβάνει το αίτημα.
5. Επαληθεύει το token αυθεντικοποίησης και ελέγχει αν ο client έχει τα απαραίτητα δικαιώματα για αυτό το endpoint.
6. Αν η επαλήθευση είναι επιτυχής, εκτελεί το ερώτημα στη βάση δεδομένων του.
7. Μορφοποιεί τα αποτελέσματα σε μορφή JSON και τα επιστρέφει στο σώμα της HTTP απάντησης (response) με κωδικό `200 OK`.

**Εναλλακτικές Ροές**:
- **A1: Εκτέλεση Ενέργειας (π.χ. Δημιουργία Κράτησης)**
  - Το εξωτερικό σύστημα θέλει να δημιουργήσει μια κράτηση για έναν υπάλληλο.
  - Δημιουργεί ένα HTTP αίτημα `POST /api/v1/bookings`.
  - Στο σώμα (body) του αιτήματος περιλαμβάνει τις λεπτομέρειες της κράτησης (user_id, resource_id, start_time, end_time) σε μορφή JSON.
  - Το Σύστημα Κρατήσεων επικυρώνει τα δεδομένα, ελέγχει τη διαθεσιμότητα και τις πολιτικές, και αν όλα είναι έγκυρα, δημιουργεί την κράτηση και επιστρέφει κωδικό `201 Created`.

**Ροές Εξαίρεσης**:
- **E1: Αποτυχία Αυθεντικοποίησης**
  - Στο βήμα 5, αν το token λείπει ή είναι άκυρο, το σύστημα απορρίπτει το αίτημα και επιστρέφει απάντηση με κωδικό `401 Unauthorized`.
- **E2: Ανεπαρκή Δικαιώματα**
  - Στο βήμα 5, αν το token είναι έγκυρο αλλά ο client δεν έχει δικαίωμα να εκτελέσει τη συγκεκριμένη ενέργεια, το σύστημα επιστρέφει απάντηση με κωδικό `403 Forbidden`.
- **E3: Λανθασμένα Δεδομένα Εισόδου**
  - Στην εναλλακτική ροή Α1, αν τα δεδομένα που στάλθηκαν στο body του request είναι ελλιπή ή λανθασμένα, το σύστημα επιστρέφει απάντηση με κωδικό `400 Bad Request` και ένα μήνυμα που περιγράφει το σφάλμα.

**Επιχειρησιακοί Κανόνες**:
- BR-10: Η πρόσβαση στο API πρέπει να ελέγχεται αυστηρά. Κάθε client πρέπει να έχει πρόσβαση μόνο στα δεδομένα και τις ενέργειες που είναι απολύτως απαραίτητες για τη λειτουργία του (Principle of Least Privilege).

**Μη-Λειτουργικές Απαιτήσεις**:
- **Ασφάλεια**: Το API πρέπει να είναι προστατευμένο από κοινές ευπάθειες (π.χ. SQL Injection, XSS). Όλη η επικοινωνία πρέπει να είναι κρυπτογραφημένη (HTTPS).
- **Τεκμηρίωση**: Το API πρέπει να διαθέτει πλήρη και σαφή τεκμηρίωση (π.χ. σε μορφή Swagger/OpenAPI) που να περιγράφει όλα τα endpoints, τις παραμέτρους, τα σχήματα δεδομένων και τα παραδείγματα.
- **Versioning**: Το API πρέπει να υποστηρίζει versioning (π.χ. /api/v1/, /api/v2/) για να επιτρέπει μελλοντικές αλλαγές χωρίς να επηρεάζονται οι υπάρχουσες ενσωματώσεις.

**Σημειώσεις**:
- Παραδείγματα χρήσης του API περιλαμβάνουν: άντληση δεδομένων χρήσης από συστήματα BI, αυτόματη κράτηση αίθουσας από ένα σύστημα ticketing, συγχρονισμός δεδομένων παρουσίας με το σύστημα HR.

### UC-08: Αναφορές και Στατιστικά Χρήσης

**Κωδικός**: UC-08

**Όνομα**: Προβολή και Εξαγωγή Αναφορών Χρήσης Χώρων

**Περιγραφή**: Το use case περιγράφει τη δυνατότητα των διαχειριστών και των υπευθύνων τμημάτων να βλέπουν συγκεντρωτικά δεδομένα, στατιστικά και αναφορές σχετικά με τη χρήση των θέσεων εργασίας και των αιθουσών. Σκοπός είναι η βελτιστοποίηση της διαχείρισης των χώρων, η κατανόηση των αναγκών και η λήψη αποφάσεων βάσει δεδομένων.

**Πρωτεύων Actor**: Διαχειριστής Συστήματος / Υπεύθυνος Τμήματος (Manager)

**Δευτερεύοντες Actors**: -

**Προϋποθέσεις**: 
- Ο χρήστης (Διαχειριστής/Manager) είναι αυθεντικοποιημένος στο σύστημα.
- Ο χρήστης έχει τα κατάλληλα δικαιώματα για να βλέπει αναφορές. Οι Managers μπορούν να βλέπουν δεδομένα μόνο για τα μέλη της ομάδας τους.

**Μετα-συνθήκες**:
- **Επιτυχής**: 
  - Ο χρήστης βλέπει τις αναφορές σε μορφή γραφημάτων και πινάκων.
  - Ο χρήστης μπορεί να εξάγει την αναφορά σε μορφή αρχείου (π.χ. CSV, Excel, PDF).
- **Αποτυχής**: 
  - Η αναφορά δεν μπορεί να δημιουργηθεί.
  - Ο χρήστης λαμβάνει μήνυμα σφάλματος.

**Βασική Ροή**:
1. Ο χρήστης επιλέγει την ενότητα "Αναφορές" από το κεντρικό μενού.
2. Το σύστημα εμφανίζει ένα dashboard με μια επισκόπηση των βασικών δεικτών (KPIs), όπως: ποσοστό πληρότητας σήμερα, συνολικός αριθμός κρατήσεων, ποσοστό no-shows.
3. Το σύστημα παρέχει μια λίστα από προκαθορισμένες αναφορές, π.χ.:
    - Πληρότητα ανά κτίριο/όροφο
    - Χρήση πόρων ανά τμήμα
    - Δημοφιλέστερες θέσεις/αίθουσες
    - Αναφορά no-show
    - Διάρκεια κρατήσεων
4. Ο χρήστης επιλέγει μια αναφορά.
5. Το σύστημα εμφανίζει φίλτρα για την παραμετροποίηση της αναφοράς, όπως χρονικό διάστημα, κτίριο, όροφος, τύπος πόρου.
6. Ο χρήστης ορίζει τα φίλτρα και πατάει "Εκτέλεση Αναφοράς".
7. Το σύστημα επεξεργάζεται τα δεδομένα και παρουσιάζει την αναφορά στην οθόνη, συνήθως με έναν συνδυασμό γραφημάτων (charts) και πινάκων (tables).
8. Ο χρήστης έχει τη δυνατότητα να πατήσει "Εξαγωγή" και να επιλέξει τη μορφή του αρχείου (π.χ. CSV).
9. Το σύστημα δημιουργεί το αρχείο και το παρέχει για λήψη (download) στον χρήστη.

**Εναλλακτικές Ροές**:
- **A1: Προγραμματισμένη Αποστολή Αναφορών**
  - Ο διαχειριστής μπορεί να ρυθμίσει την αυτόματη δημιουργία και αποστολή συγκεκριμένων αναφορών μέσω email σε προκαθορισμένα χρονικά διαστήματα (π.χ. εβδομαδιαία αναφορά πληρότητας).

**Ροές Εξαίρεσης**:
- **E1: Δεν υπάρχουν δεδομένα**
  - Στο βήμα 7, αν δεν υπάρχουν δεδομένα που να ταιριάζουν στα κριτήρια του χρήστη, το σύστημα εμφανίζει ένα ενημερωτικό μήνυμα.

**Επιχειρησιακοί Κανόνες**:
- BR-11: Οι Managers έχουν πρόσβαση μόνο σε ανωνυμοποιημένα ή συγκεντρωτικά δεδομένα για λόγους προστασίας προσωπικών δεδομένων, εκτός αν αφορά άμεσα τα μέλη της ομάδας τους.

**Μη-Λειτουργικές Απαιτήσεις**:
- **Απόδοση**: Η δημιουργία των περισσότερων αναφορών δεν πρέπει να ξεπερνά τα 15 δευτερόλεπτα.
- **Ακρίβεια Δεδομένων**: Οι αναφορές πρέπει να αντικατοπτρίζουν με ακρίβεια τα δεδομένα που υπάρχουν στη βάση του συστήματος.

**Σημειώσεις**:
- Τα δεδομένα από τις αναφορές μπορούν να χρησιμοποιηθούν για τον εντοπισμό χώρων που δεν χρησιμοποιούνται αρκετά (και μπορεί να αλλάξουν χρήση) ή περιοχών με υπερβολική ζήτηση (όπου μπορεί να χρειάζονται περισσότερες θέσεις).

### UC-09: Διαχείριση Πολιτικών Κράτησης και Ειδοποιήσεων

**Κωδικός**: UC-09

**Όνομα**: Καθορισμός και Εφαρμογή Πολιτικών Κράτησης, Ειδοποιήσεων και Διαχείρισης Μη Εμφάνισης (No-Show)

**Περιγραφή**: Το use case περιγράφει τη διαδικασία με την οποία ο διαχειριστής του συστήματος ορίζει ένα σύνολο κανόνων (πολιτικών) που διέπουν τις κρατήσεις, ρυθμίζει τις αυτόματες ειδοποιήσεις προς τους χρήστες και διαχειρίζεται τις περιπτώσεις όπου ένας χρήστης κάνει κράτηση αλλά δεν εμφανίζεται (no-show).

**Πρωτεύων Actor**: Διαχειριστής Συστήματος

**Δευτερεύοντες Actors**: 
- Σύστημα Κρατήσεων (ως το σύστημα που επιβάλλει τις πολιτικές)
- Υπάλληλος (ως παραλήπτης των ειδοποιήσεων και υποκείμενο των πολιτικών)

**Προϋποθέσεις**: 
- Ο διαχειριστής είναι αυθεντικοποιημένος στο διαχειριστικό περιβάλλον.

**Μετα-συνθήκες**:
- **Επιτυχής**: 
  - Οι πολιτικές κρατήσεων έχουν οριστεί και εφαρμόζονται σε όλες τις νέες κρατήσεις.
  - Οι ειδοποιήσεις αποστέλλονται αυτόματα στα σωστά χρονικά σημεία.
  - Οι κρατήσεις που χαρακτηρίζονται ως no-show ακυρώνονται αυτόματα.
- **Αποτυχής**: 
  - Οι ρυθμίσεις δεν αποθηκεύονται.

**Βασική Ροή (Ορισμός Πολιτικών)**:
1. Ο διαχειριστής επιλέγει την ενότητα "Πολιτικές Κρατήσεων" από το μενού.
2. Το σύστημα εμφανίζει μια φόρμα με τις διαθέσιμες ρυθμίσεις, ομαδοποιημένες σε κατηγορίες:
    - **Πολιτικές Κράτησης**: 
        - Μέγιστος αριθμός ημερών για προ-κράτηση (π.χ. 14 ημέρες).
        - Μέγιστη διάρκεια μιας κράτησης (π.χ. 8 ώρες για θέση, 4 ώρες για αίθουσα).
        - Μέγιστος αριθμός ενεργών κρατήσεων ανά χρήστη.
        - Χρονικό όριο ακύρωσης από τον χρήστη (π.χ. τουλάχιστον 1 ώρα πριν την έναρξη).
    - **Διαχείριση No-Show**:
        - Ενεργοποίηση/Απενεργοποίηση της πολιτικής no-show.
        - Χρονικό παράθυρο για check-in (π.χ. ο χρήστης πρέπει να κάνει check-in από 15 λεπτά πριν έως 15 λεπτά μετά την έναρξη της κράτησης).
        - Αυτόματη ενέργεια σε περίπτωση no-show: Ακύρωση της κράτησης και απελευθέρωση του πόρου.
        - (Προαιρετικά) Σύστημα ποινών: Μετά από Χ no-shows, ο χρήστης μπορεί να χάσει προσωρινά το δικαίωμα κράτησης.
    - **Ειδοποιήσεις**:
        - Ενεργοποίηση/Απενεργοποίηση ειδοποιήσεων.
        - Ρύθμιση προτύπων (templates) για κάθε τύπο ειδοποίησης (email και push notification): Επιβεβαίωση Κράτησης, Υπενθύμιση Κράτησης (π.χ. 1 ώρα πριν), Ειδοποίηση Ακύρωσης, Προειδοποίηση για No-Show.
3. Ο διαχειριστής τροποποιεί τις τιμές στα πεδία σύμφωνα με την πολιτική της εταιρείας.
4. Ο διαχειριστής αποθηκεύει τις αλλαγές.
5. Το Σύστημα Κρατήσεων εφαρμόζει πλέον αυτές τις πολιτικές σε όλες τις σχετικές λειτουργίες (δημιουργία κράτησης, ακύρωση, check-in).

**Εναλλακτικές Ροές**:
- **A1: Εφαρμογή Πολιτικής No-Show**
  - Ένας χρήστης έχει μια κράτηση που ξεκινά στις 09:00.
  - Το χρονικό παράθυρο για check-in λήγει στις 09:15.
  - Στις 09:16, το σύστημα ελέγχει την κατάσταση της κράτησης. Αν δεν έχει γίνει check-in (ούτε αυτόματα μέσω RFID, ούτε χειροκίνητα), το σύστημα:
    1. Αλλάζει την κατάσταση της κράτησης σε "No-show".
    2. Ακυρώνει την κράτηση, απελευθερώνοντας τον πόρο.
    3. Στέλνει ειδοποίηση στον χρήστη ότι η κράτησή του ακυρώθηκε λόγω μη εμφάνισης.
    4. Αυξάνει τον μετρητή no-show για τον συγκεκριμένο χρήστη.

**Ροές Εξαίρεσης**: -

**Επιχειρησιακοί Κανόνες**:
- BR-12: Οι πολιτικές μπορούν να οριστούν καθολικά ή να είναι διαφορετικές ανάλογα με τον τύπο του πόρου (π.χ. διαφορετικοί κανόνες για θέσεις και διαφορετικοί για αίθουσες).

**Μη-Λειτουργικές Απαιτήσεις**:
- **Ευελιξία**: Το σύστημα πρέπει να είναι αρκετά ευέλικτο ώστε να επιτρέπει τη διαμόρφωση πολλαπλών και σύνθετων κανόνων.

**Σημειώσεις**:
- Η σωστή ρύθμιση αυτών των πολιτικών είναι θεμελιώδης για την ομαλή και δίκαιη λειτουργία του συστήματος hot-desking.
